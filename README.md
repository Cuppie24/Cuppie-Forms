# Cuppie Forms
Система управления формами с полноценной аутентификацией пользователей, построенная на принципах Clean Architecture.

## Архитектура

Проект реализован с использованием принципов **Clean Architecture**, что обеспечивает четкое разделение ответственности и высокую тестируемость кода.

### Структура проекта

```
├── compose.yaml
├── cuppie_forms_backend/
│   ├── src/
│   │   ├── Cuppie.Api/                    # Слой представления (Controllers, Middleware)
│   │   ├── Cuppie.Infrastructure/         # Слой инфраструктуры
│   │   │   └── Data/                      # Контекст БД и миграции
│   │   │       └── Migrations/
│   │   ├── Cuppie.Application/            # Слой бизнес-логики
│   │   ├── Cuppie.Domain/                 # Слой домена (Entities, Interfaces)
│   │   └── UnitTests/                     # Модульные тесты
│   ├── Dockerfile
│   └── cuppie_forms_api.sln
└── cuppie_forms_frontend/
    ├── vite.config.ts
    └── Dockerfile
```

### Слои архитектуры

- **Domain** - Содержит бизнес-сущности и интерфейсы
- **Application** - Бизнес-логика, DTO, сервисы приложения
- **Infrastructure** - Реализация доступа к данным, внешние сервисы
- **API** - Контроллеры, middleware, конфигурация

## Основные компоненты

### Система аутентификации

Реализована полноценная система аутентификации с использованием JWT токенов:

#### AuthService
- **Регистрация пользователей** - создание новых аккаунтов с валидацией уникальности
- **Аутентификация** - проверка учетных данных и выдача JWT токенов
- **Обновление токенов** - механизм refresh токенов
- **Извлечение данных пользователя** - получение информации из JWT

#### CryptoService
- **Хеширование паролей** - используется SHA256 с солью
- **Генерация соли** - криптографически стойкая соль для каждого пароля
- **Верификация паролей** - проверка введенного пароля с сохраненным хешем

#### JwtTokenService
- **Генерация JWT токенов** - создание подписанных токенов с claims
- **Валидация токенов** - проверка подписи и срока действия
- **Refresh механизм** - обновление просроченных токенов
- **Извлечение claims** - получение данных пользователя из токена

### Особенности реализации

#### Безопасность
- Пароли хешируются с использованием SHA256 и уникальной соли
- JWT токены подписываются симметричным ключом HMAC-SHA256
- Валидация токенов включает проверку подписи, срока действия и алгоритма

#### Обработка ошибок
- Использование паттерна `OperationResult<T>` для централизованной обработки ошибок
- Структурированные коды ошибок (Conflict, Unauthorized, InternalServerError)
- Подробное логирование всех операций с использованием Serilog

#### Логирование
- Интеграция с Serilog для структурированного логирования
- Логирование критических операций (регистрация, аутентификация, обновление токенов)
- Детальная информация об ошибках с трассировкой стека

## Быстрый старт

### 1. Установка зависимостей

```bash
dotnet restore
```

### 2. Настройка базы данных
В файле `compose.yaml` в блоке `db` заменить на свои данные (любые)
```yaml
    db:
    image: postgres:16
    restart: always
    environment:
      POSTGRES_DB: <имя вашей базы данных>
      POSTGRES_USER: <ваше имя пользователя>
      POSTGRES_PASSWORD: <ваш пароль>
    ports:
      - "5432:5432" // в дальнейшем удалим для безопасности
    container_name: db
    volumes:
      - pgdata:/var/lib/postgresql/data
```
После этого обновить строку подключения в файле `appsettings.json`, который находится по пути `cuppie_forms_backend/src/Cuppie.Api`
```
"ConnectionStrings": {
    "cuppieContext": "Host=localhost;Port=5432;Database=<имя вашей базы данных>;Username=<ваше имя пользователя>;Password=<ваш пароль>"
  },
```

Создайте миграцию:

```bash
dotnet ef migrations add InitialCreate \
  --project ./src/Cuppie.Infrastructure \
  --startup-project ./src/Cuppie.Api \
  --context CuppieDbContext \
  --output-dir Data/Migrations
```

Обновите базу данных:

```bash
dotnet ef database update \
  --project ./src/Cuppie.Infrastructure \
  --startup-project ./src/Cuppie.Api \
  --context CuppieDbContext
```

Если после этого вы больше не собираетесь работать над БД удалите строки `ports` из compose.yaml в блоке `db` чтобы к базе данных не было доступа извне
```
db:
    ports:          //  удалить
      - "5432:5432" //  удалить
```
Также в строке подключение заменить `localhost` на `db`
```
"ConnectionStrings": {
    "cuppieContext": "Host=db;Port=5432;Database=<имя вашей базы данных>;Username=<ваше имя пользователя для базы данных>;Password=<ваш пароль базы данных>"
  },
```

### 3. Запуск приложения

```bash
docker compose up --build -d
```

После запуска приложение будет доступно по адресу: **http://localhost:3000**

## Конфигурация портов

### Backend (API)

Для изменения портов бэкенда отредактируйте файл `compose.yaml`:

```yaml
services:
  cuppie_forms_backend:
    environment:
      - ASPNETCORE_URLS=https://+:5001;http://+:5000
```
После этого в Frontend тоже надо изменить адрес для отправки запросов. Для этого в файле `cuppie_forms_frontend/.env` нужно указать правильный адрес с портом
```
VITE_API_URL=http://localhost:5000/api
```
### Frontend

Для изменения порта фронтенда:

1. Отредактируйте файл `vite.config.ts`:
```typescript
export default defineConfig({
  server: {
    port: 3000  // Измените на нужный порт
  }
})
```

2. Обновите `compose.yaml` для проброса нужных портов
```yaml
ports:
  5000:5000
  5001:5001
```
3. Обновите `Dockerfile` для открытия нового порта
```Docker
EXPOSE 5000
EXPOSE 5001
```


## Устранение неполадок

### Проблемы с базой данных

Если миграция не применяется:
1. Убедитесь, что база данных запущена
2. Проверьте строку подключения в конфигурации
3. Удалите старые миграции и создайте новые

### Проблемы с Docker

Если контейнеры не запускаются:
1. Проверьте, что Docker запущен
2. Убедитесь, что порты не заняты другими приложениями
3. Попробуйте пересобрать образы: `docker compose build --no-cache`
